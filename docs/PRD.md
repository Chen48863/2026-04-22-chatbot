# 產品需求文件（PRD）

### 1. 概述（Overview）
- **產品名稱**：智慧聊天機器人（Smart Chatbot）
- **版本**：1.0.0
- **日期**：2026-04-22
- **作者**：Antigravity

### 2. 目標與背景（Goals & Background）
- **目標**：建立一個功能完善的聊天機器人 Web 應用程式，支援多輪對話、多聊天室管理、檔案上傳、對話歷史管理、回答控制與記憶機制。
- **背景**：隨著 AI 聊天應用的普及，使用者需要一個具備現代化介面與完整功能的聊天平台。本產品旨在提供類似 ChatGPT 的使用體驗，讓使用者能夠方便地與 AI 互動。
- **痛點與需求**：
  - 現有簡易聊天工具缺乏對話歷史管理功能
  - 無法上傳檔案作為對話參考
  - 缺少重新生成回答或中止回應的控制能力
  - 無法記住使用者偏好，每次對話需重新設定

### 3. 使用者故事（User Stories）
1. 作為**一般使用者**，我希望**建立新的聊天室**，以便**針對不同主題分開對話**。
2. 作為**一般使用者**，我希望**在聊天室中輸入訊息並獲得回覆**，以便**與 AI 進行多輪對話**。
3. 作為**一般使用者**，我希望**上傳圖片或文件作為對話參考**，以便**讓 AI 根據附件內容回答問題**。
4. 作為**一般使用者**，我希望**切換到過去的聊天記錄**，以便**繼續之前未完成的對話**。
5. 作為**一般使用者**，我希望**重新生成 AI 的回答**，以便**獲得更好或不同的回覆**。
6. 作為**一般使用者**，我希望**中止 AI 正在生成的回答**，以便**節省等待時間**。
7. 作為**一般使用者**，我希望**系統記住我的偏好設定**，以便**在不同對話中保持一致的使用體驗**。

### 4. 功能需求（Functional Requirements）

#### F-01：對話狀態管理（P0 必要）
- 支援建立多個聊天室（Session）
- 每個聊天室維持獨立的上下文
- 可為聊天室命名與重新命名
- 可刪除不需要的聊天室
- **驗收標準**：使用者可建立至少 10 個聊天室，切換時各自的對話內容獨立不混淆

#### F-02：訊息系統（P0 必要）
- 訊息結構包含：角色（role: user/assistant）、內容（content）、時間戳記（timestamp）
- 支援文字訊息的發送與接收
- 訊息依時間順序排列顯示
- **驗收標準**：每則訊息正確顯示角色、內容與時間

#### F-03：上傳圖片或文件（P1 重要）
- 支援上傳圖片檔案（jpg、png、gif）
- 支援上傳文件檔案（pdf、txt、docx）
- 上傳的檔案顯示在對話中
- 檔案大小限制：單檔最大 10MB
- **驗收標準**：使用者可上傳支援格式的檔案，檔案正確顯示在聊天視窗中

#### F-04：對話歷史管理（P0 必要）
- 側邊欄顯示所有聊天室列表
- 可點擊切換至不同聊天室
- 顯示聊天室的最後訊息預覽
- 依照最後更新時間排序
- **驗收標準**：側邊欄正確顯示所有聊天室，點擊後載入完整對話歷史

#### F-05：回答控制（P1 重要）
- 提供「重新生成」按鈕，可重新產生 AI 的最後回答
- 提供「中止生成」按鈕，可在 AI 回答生成中中止
- **驗收標準**：重新生成產生不同的回覆內容；中止功能可在回答生成中立即停止

#### F-06：記憶機制（P1 重要）
- 儲存使用者設定的顯示名稱
- 儲存使用者的偏好設定（如：回覆語言、回覆風格）
- 記憶資訊跨聊天室持續有效
- **驗收標準**：設定偏好後，新建聊天室仍保持相同設定

### 5. 非功能需求（Non-Functional Requirements）
- **效能需求**：頁面載入時間 < 2 秒；訊息發送後 < 3 秒顯示回覆
- **安全性需求**：上傳檔案需驗證格式與大小；防止 XSS 攻擊
- **可擴展性需求**：架構設計應便於日後替換 Mock 為真實 AI API
- **相容性需求**：支援 Chrome、Firefox、Edge 最新版本；支援行動裝置瀏覽

### 6. 系統架構（System Architecture）
- **前端**：HTML + CSS + JavaScript，使用 Jinja2 模板引擎
- **後端**：FastAPI（Python 非同步 Web 框架）
- **資料庫**：SQLite（使用 SQLAlchemy ORM）
- **AI 回覆**：Mock 模擬回應（預留 AI API 串接介面）

### 8. 風險評估（Risk Assessment）
| 風險 | 影響 | 緩解策略 |
|------|------|----------|
| Mock 回應與真實 AI 行為差異 | 上線後需調整介面 | 設計抽象化的 AI 回覆介面，方便日後替換 |
| 檔案上傳的安全風險 | 惡意檔案攻擊 | 嚴格驗證檔案格式與大小限制 |
| SQLite 效能瓶頸 | 大量資料時查詢變慢 | 設計索引，預留資料庫遷移方案 |

- 頁面載入時間 < 2 秒
