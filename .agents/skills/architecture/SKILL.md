# Architecture (軟體架構文件) Skill

## 描述

此技能用於生成軟體架構文件（Software Architecture Document）。當使用者要求撰寫架構文件時，AI Agent 應依照以下結構產出完整的架構設計文件。

## 觸發條件

當使用者提出以下類型的請求時啟用此技能：
- 「/architecture」
- 「幫我寫架構文件」
- 「產生系統架構設計」
- 「建立架構文件」

## 輸出格式

請依照以下模板撰寫架構文件：

### 1. 概述（Overview）
- **系統名稱**：[名稱]
- **版本**：[版本號]
- **日期**：[撰寫日期]
- **作者**：[作者名稱]
- **文件目的**：描述此架構文件的目的與適用範圍

### 2. 架構目標與約束（Goals & Constraints）
- **設計原則**：列出系統遵循的核心設計原則（如：單一職責、鬆耦合、高內聚）
- **技術限制**：現有技術環境的限制條件
- **合規要求**：需遵循的法規或標準（如適用）
- **假設與前提**：架構設計所基於的假設

### 3. 系統架構圖（System Architecture Diagram）
- 高層級系統架構圖（建議使用 C4 模型或 UML）
- 系統邊界與外部系統的互動
- 主要元件之間的關係與通訊方式
- 可使用 Mermaid 語法繪製架構圖

### 4. 元件設計（Component Design）
- 列出每個主要元件/模組：
  - **元件名稱**：[名稱]
  - **職責**：該元件負責的功能
  - **介面**：對外提供的介面定義
  - **依賴關係**：與其他元件的依賴
  - **技術選型**：使用的技術/框架

### 5. 資料架構（Data Architecture）
- **資料模型**：核心資料實體與其關係（ER Diagram）
- **資料庫設計**：使用的資料庫類型與設計考量
- **資料流**：資料在系統中的流動路徑
- **資料儲存策略**：快取、持久化、備份等策略

### 6. API 設計（API Design）
- **對外 API**：
  - 端點（Endpoint）
  - HTTP 方法
  - 請求/回應格式（含範例）
  - 認證方式
- **對內 API / 元件間通訊**：
  - 通訊協定（REST、gRPC、訊息佇列等）
  - 介面定義

### 7. 技術堆疊（Technology Stack）
| 層級 | 技術選擇 | 說明 |
|------|----------|------|
| 前端 | [技術] | [選擇原因] |
| 後端 | [技術] | [選擇原因] |
| 資料庫 | [技術] | [選擇原因] |
| 雲端/基礎設施 | [技術] | [選擇原因] |
| CI/CD | [技術] | [選擇原因] |
| 監控 | [技術] | [選擇原因] |

### 8. 部署架構（Deployment Architecture）
- **部署環境**：開發、測試、正式環境的配置
- **容器化策略**：Docker / Kubernetes 配置（如適用）
- **CI/CD 流程**：自動化建構、測試、部署流程
- **基礎設施即代碼（IaC）**：使用的 IaC 工具與配置

### 9. 安全架構（Security Architecture）
- **認證機制**：使用者身份驗證方式（OAuth2、JWT 等）
- **授權機制**：角色與權限管理（RBAC、ABAC 等）
- **資料加密**：傳輸中加密（TLS）與靜態加密
- **安全策略**：防火牆、WAF、DDoS 防護等
- **敏感資料處理**：密鑰管理、個資保護

### 10. 非功能需求實現（NFR Implementation）
- **效能**：預期的回應時間、吞吐量，以及如何達成
- **可擴展性**：水平/垂直擴展策略
- **可用性**：高可用設計（HA）、容錯機制
- **可維護性**：程式碼規範、模組化設計
- **可觀測性**：日誌、指標、追蹤（Observability 三支柱）

### 11. 整合與第三方服務（Integration）
- 列出所有需整合的外部系統或第三方服務：
  - **服務名稱**：[名稱]
  - **整合方式**：API / SDK / Webhook 等
  - **用途**：該整合的目的
  - **失敗處理**：整合失敗時的降級策略

### 12. 錯誤處理與監控（Error Handling & Monitoring）
- **錯誤處理策略**：全域錯誤處理、重試機制、斷路器模式
- **日誌策略**：日誌等級、格式、集中化管理
- **告警機制**：告警條件、通知方式
- **健康檢查**：系統健康狀態監控端點

### 13. 架構決策紀錄（Architecture Decision Records, ADR）
- 針對每個重要的架構決策：
  - **決策標題**：[標題]
  - **背景**：為什麼需要做這個決策
  - **選項分析**：考慮過的方案與各自的優缺點
  - **最終決策**：選擇的方案及其理由
  - **影響**：這個決策帶來的影響與後果

## 注意事項

- 所有架構文件內容應以**繁體中文**撰寫
- 架構圖建議使用 **Mermaid** 語法，方便直接在 Markdown 中呈現
- 每個元件的描述應清晰明確，避免模糊用語
- 技術選型需附上**選擇原因**，而非僅列出技術名稱
- 如有尚未確定的設計決策，應標註為「待確認」
- 文件應保持**活文件（Living Document）**的精神，隨專案演進持續更新
